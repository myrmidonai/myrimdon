## 1. 产品定位

### 1.1 是什么

Myrmidon 是一个**通用自治工作流运行时**。

它持续维护一个"期望的工程世界"与"真实世界"之间的一致性，并通过协调 AI Agent 和人类协作者来完成这个目标。

适用于任何需要多步骤、多角色、长周期协作的工作流场景：

| 领域 | 示例工作流 |
|------|-----------|
| 软件开发 | 需求 → 设计 → 编码 → QA → 交付（内置模板） |
| 内容创作 | 选题 → 脚本 → 审核 → 发布 |
| 产品运营 | 选品分析 → 上架 → 定价 → 复盘 |
| 内容审核 | 摄取 → AI 粗筛 → 人工复核 → 决策 |

### 1.2 不是什么

- 不是 AI Chat 工具
- 不是 Copilot / 代码补全工具
- 不是普通 Multi-Agent Framework
- 不是 Workflow Builder（可视化连线工具）
- 不是 CI/CD 流水线（虽然形式上类似）

### 1.3 核心区别：Runtime vs Pipeline

CI/CD Pipeline 追问：**我们执行到第几步了？**

Myrmidon 追问：**世界现在是我期望的样子吗？**

这一字之差决定了系统在长期运行中是否能保持稳定。

### 1.4 与 PRD1/PRD2 的关系

PRD1 描述了软件开发专用编排器。PRD2 将其扩展为通用工作流引擎。PRD3 在此基础上引入**世界一致性维护**作为系统核心范式，并建立稳健的自治运行模型。

PRD1/PRD2 中所有产品功能定义（CLI 命令、DSL、TUI、软件开发模板）均继承有效，PRD3 在其上补充架构原则和稳定性设计。

## 2. 核心哲学

### 原则 1 — Runtime 是世界状态的唯一拥有者

任何 Agent Worker：
- **无权**声明任务完成
- **无权**直接修改系统状态
- **无权**决定验收结果

Agent 仅能：提议修改、生成产物、局部执行。

所有状态变更由 Runtime 裁决后写入。

---

### 原则 2 — Artifact 是唯一真相

世界的真实状态由工程产物构成：

- 代码、设计稿、原型
- 页面截图、测试结果、构建产物
- 验证报告、QA 报告

**Memory 不是真相。Summary 不是真相。Agent 的说法不是真相。**

当 Runtime 状态与 Artifact 现实不一致时，以 Artifact 为准。

---

### 原则 3 — 验证决定完成，Human 是一等公民 Validator

任务是否完成，只能由验证系统决定。

验证分三类：

| 类型 | 说明 | 可信度 |
|------|------|--------|
| **自动化验证** | 编译、测试、lint、构建 | 高，完全可信 |
| **AI 辅助验证** | 截图对比设计稿、视觉结构分析 | 中，辅助参考 |
| **人工验证** | 人类确认 UI 语义、交付物质量 | 最终权威 |

人工验证不是"审批门控的附属品"，而是验证系统的一等公民。**对于无法完全自动化的 artifact（尤其是 UI/UX），人工验证是 valid 状态的必要条件。**

系统目标不是消灭人工验证，而是让人工验证变少、变快、有据可查。

---

### 原则 4 — 持续对账，而非边界校验

系统必须持续检测：**Runtime 的认知** 与 **Artifact 的现实** 是否一致。

不是在节点完成时验证一次，而是持续问：

> "我以为有效的东西，现在还有效吗？"

当上游 artifact 发生变化，所有依赖它的下游 artifact **自动标记为 stale**，下游节点自动暂停。

注意：stale 标记不触发自动重执行，而是等待调度决策。这避免了级联重跑的资源浪费。

---

### 原则 5 — Worker 是无状态的瞬时执行单元

所有 Agent Worker：
- 无长期记忆，无持久上下文
- 每次执行注入**最小必要上下文**（仅当前任务所需）
- 执行结束立即销毁
- 可被任意替换或重建

系统不能依赖长 session、长 memory、长上下文。这是对抗 AI 认知漂移的核心机制。

---

### 原则 6 — 有界自治，失败必须收敛

系统**不追求无限重试直到成功**。每次失败必须带来信息增益，循环必须收敛。

**防卡死的三个机制：**

1. **有界重试**：每个节点设定最大尝试次数，耗尽后自动暂停等待人工指导。

2. **结构化反馈注入**：人工拒绝时必须提供结构化原因（哪里错了 + 期望是什么）。该反馈成为下次 Agent 上下文的一部分。下一次不是"重试"，而是"带着纠正信息重试"。

3. **升级路径**：失败 1 次 → 自动重试；失败 N 次 → 通知人类；失败 N+M 次 → 暂停并要求明确指导。

## 3. 产品形态

### 3.1 运行模式

Myrmidon 以**持续运行的后台进程**存在，不依附于某次命令调用的生命周期。

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **托盘应用**（默认） | 关闭窗口后最小化到系统托盘，Runtime 继续运行 | 本地开发机 |
| **Daemon 模式** | 纯后台服务，无 UI | 服务器、CI 环境 |

两种模式共享同一个 Runtime 进程。CLI 命令向 Runtime 发送指令，不直接是 Runtime 本身。

`myrmidon start --workflow X` = 向 Runtime 发起工作流，不是启动新进程。

### 3.2 CLI 界面

继承 PRD2.md 定义的完整 CLI 命令体系，新增：

```
myrmidon daemon start/stop/status   # Daemon 模式管理
myrmidon status                      # 查看当前所有 workflow 状态（Runtime 进程存活时）
myrmidon review <run-id>             # 进入人工审核队列
```

### 3.3 TUI 界面

继承 PRD2.md 定义的 5-Tab TUI，核心视图：

- **Overview**：当前活跃工作流的 Artifact 有效性状态图（非"步骤进度"）
- **Review Queue**：待人工验证的 artifact 列表，含对比视图
- **Logs**：事件流 + Reconciliation 日志
- **Cron**：定时器与调度状态

### 3.4 通知与 IM 集成

当工作流需要人工介入时，系统通过配置的通知渠道推送：

- 托盘通知（本地）
- Slack / 企业微信 / Email（远程）

通知包含：当前等待的决策类型、上下文摘要、可直接回复的快捷操作。

## 4. 工作流系统

继承 PRD2.md 定义的完整工作流系统，以下为核心概念概述。

### 4.1 工作流定义

用户通过 TypeScript DSL（`defineWorkflow()`）描述工作流有向图，引擎按 DAG 执行。工作流结构在启动前静态确定，Runtime 按状态机推进，**不允许 Agent 自主决定流转**。

### 4.2 内置节点类型

| 类型 | 说明 |
|------|------|
| `agent` | 派发给 AI Worker 执行 |
| `human_approval` | 人工验证门控（Human Validator） |
| `condition` | 按条件选择出边 |
| `parallel_fork` | 并行启动多个分支 |
| `join` | 等待所有并行分支完成 |
| `transform` | 纯函数转换产物 |
| `trigger` | 工作流入口（手动/定时/Webhook） |
| `loop` | 循环子图直到条件满足 |

### 4.3 内置模板

- `software-dev-agile`：敏捷软件开发（需求→设计→编码→QA→交付）
- `software-dev-waterfall`：瀑布式，每阶段含人工门控

用户可 fork 内置模板自定义。

## 5. Artifact 世界

### 5.1 Artifact 是系统核心

系统跟踪的不是"哪些节点跑完了"，而是"哪些产物是有效的"。

工作流进度 = 期望 artifact 集合中，有多少已处于 `valid` 状态。

### 5.2 Artifact 生命周期状态

```
pending → generating → needs_validation → valid
                                        ↘ stale
                                        ↘ invalid
                                        ↘ needs_review   ← 等待 Human Validator
```

### 5.3 Artifact 依赖与 Stale 传播

每个 artifact 声明它依赖哪些上游 artifact。当上游发生变化：

1. 所有直接和间接依赖该 artifact 的下游产物**自动标记为 stale**
2. 下游相关节点自动暂停
3. **不自动触发重执行**，由调度器或人工决策是否重建

这防止了大规模级联重跑，同时保证系统不会在 stale 产物上继续推进。

### 5.4 系统可从 Artifact 重建

删除所有 session、memory、摘要后，系统必须能从：

- 文件系统上的 artifact 内容
- 事件日志

重构完整的 workflow 运行状态，并继续执行。这是 Runtime 稳定性的基础保障。

## 6. 验证系统

### 6.1 验证分层

**工程验证**（完全自动化，高可信）
- 编译 / 类型检查
- 单元测试 / 集成测试
- 依赖合法性
- 构建产物完整性

**UI 验证**（分层，诚实对待自动化边界）
- Structural：页面结构、组件层级存在性
- Design Token：颜色、字体、间距是否符合设计规范
- Responsive：响应式断点行为
- Interaction：交互状态（hover, focus, transition）
- **Semantic（语义层）**：视觉层级是否合理、信息组织是否符合预期 — 此层目前**只能由人工验证**，系统不追求自动化

**治理验证**（规则化）
- ADR 合规性
- 架构约束
- 项目策略检查

### 6.2 验证诚实性原则

系统设计必须显式承认：

> **对于 UI/UX artifact，100% 自动化验证在现阶段不可能做到。**

这不是系统缺陷，而是现实约束。系统的响应是：

- 自动化验证负责能验证的部分（结构、token、交互逻辑）
- Human Validator 负责语义层判断
- 每次人工验证的结论（通过/拒绝 + 原因）被记录为 artifact 的一部分，成为后续 Agent 上下文

### 6.3 Human Validator 工作界面

当 artifact 进入 `needs_review` 状态，系统为人类提供：

- 设计稿 vs 实现截图的并排对比
- Design token 差异高亮
- 上一次拒绝原因（如有）
- 快捷操作：通过 / 拒绝（必须填写具体原因） / 推迟

拒绝时填写的结构化原因自动注入下次 Agent 执行上下文，使循环收敛。

## 7. Reconciliation 引擎

### 7.1 持续对账循环

系统后台持续运行对账循环，检测：

- Runtime 认为 valid 的 artifact，文件系统上是否真实存在且内容未变
- 依赖关系是否因上游变化而失效
- 孤立的 worktree（执行已结束但资源未释放）
- 长时间停留在 `running` 状态的节点（可能是 crash 遗留）
- 人工审核超时未响应的节点

### 7.2 漂移类型与处理

| 漂移类型 | 触发条件 | 处理方式 |
|---------|---------|---------|
| Missing Artifact | 文件消失 | 标 invalid，通知 |
| Stale Artifact | 上游 checksum 变化 | 标 stale，暂停下游 |
| Phantom Running | 节点 running 超时无心跳 | 标 failed，触发重试策略 |
| Orphan Worktree | 执行结束但 worktree 残留 | 自动清理 |
| Invalid Projection | 文档说完成但验证失败 | 标 invalid，重新入队 |

### 7.3 对账不是扫描全部

对账采用 dirty-bit + 事件驱动机制：
- 变化发生时立即传播 stale 标记（事件驱动，快速）
- 定期全量 checksum 扫描作为兜底（周期性，慢速）

世界静止时，对账几乎零开销。

## 8. 有界自治模型

### 8.1 重试策略

每个节点配置：
- 最大自动重试次数
- 重试间隔策略（递增）
- 耗尽后行为：`pause_for_human`（不是 abort）

### 8.2 失败升级路径

```
第 1 次失败 → 自动重试
第 N 次失败 → IM 通知人类，继续等待
第 N+M 次失败 → 暂停工作流，要求人类提供明确指导才能继续
```

### 8.3 结构化反馈循环

人类拒绝一个 artifact 时，必须提供结构化反馈：

```
拒绝原因分类：
  - layout_wrong       (哪个区域，期望是什么)
  - token_mismatch     (哪个属性，期望值是什么)
  - logic_error        (哪段逻辑，正确行为是什么)
  - requirement_gap    (缺少什么功能或内容)
  - other              (自由文本)
```

这些反馈与 artifact 绑定存储，作为下次 Agent 执行的精准上下文。

**结果**：每个循环都在收敛，不是原地旋转。

### 8.4 现实预期

当前技术条件下，对于完整软件开发工作流：

- **约 70% 的工作**可完全自主完成（后端逻辑、API、测试、文档）
- **约 30% 的工作**需要人工介入（UI 语义验收、架构决策、需求澄清）

系统的价值不在于消灭这 30%，而在于：**让这 30% 的人工介入高效、有依据、有上下文。**

## 9. Human Governance 模型

### 9.1 人类角色重定义

人类在 Myrmidon 中的角色是**校准者（Calibrator）**，不是**执行审批者（Approver）**。

区别：
- Approver：等系统完成了来点头
- Calibrator：设定约束和标准，处理系统无法自主决策的例外，每次介入都使系统下次更准确

### 9.2 人类负责的决策

| 决策类型 | 说明 |
|---------|------|
| 产品方向 | 需求确认、目标定义 |
| 架构决策（ADR） | 技术选型、重要架构选择，一经确认即为不可变治理产物 |
| UI 语义验收 | 视觉/体验层最终确认 |
| 高风险仲裁 | 系统达到重试上限、发现架构冲突等需要人工判断的情况 |

### 9.3 人类不负责的事情

- 重复性工程执行
- 每一步的手动触发
- 格式化、样板代码、文档生成
- 构建/测试/部署的日常运行

### 9.4 审核队列

以下情况自动进入人工审核队列：

- artifact 进入 `needs_review` 状态（Human Validator 触发）
- 节点达到最大重试次数
- Reconciliation 检测到无法自动修复的漂移
- 熵增指标超过阈值（系统认为当前状态不可靠）

## 10. 可靠性模型

### 10.1 系统必须支持

| 能力 | 含义 |
|------|------|
| **Replay** | 从事件日志重放，重建任意历史状态 |
| **Recovery** | 崩溃后重启，从持久化状态继续执行 |
| **Rollback** | 将 artifact 回退到之前的 checkpoint |
| **Rebuild** | 删除所有 session/memory/摘要后，从 artifact 重建 Runtime 状态 |
| **Isolation** | 每个 Worker 执行不影响其他 Worker 和 Runtime 状态 |
| **Cleanup** | 自动回收孤立 worktree、临时资源 |

### 10.2 崩溃恢复语义

Runtime 重启后：

1. 读取持久化的事件日志和 artifact 状态
2. 对所有标记为 `running` 的节点执行 stale 检查（crash 前可能部分完成）
3. 对孤立的 `running` 状态（无心跳记录）标记为 `failed`，触发重试策略
4. 恢复 Reconciliation 循环
5. 继续执行

用户看到：系统重启后自动从中断处继续，无需手动干预。

### 10.3 长期稳定性保证

系统长期运行后**不应发生**：

- 世界状态漂移（Runtime 认为的 ≠ 磁盘上的）
- 隐藏状态污染（过期认知影响后续决策）
- 不可恢复的上下文腐化
- Worktree 泄漏（孤立的 git worktree 堆积）
- 幻觉完成（artifact 未真正 valid 但系统认为完成）

## 11. 成功指标

### 工程指标
- 自主完成率（无人工介入的节点占比）
- 验证通过率（首次提交即通过验证的 artifact 占比）
- Reconciliation 准确率（检测到的漂移 vs 实际漂移）
- 崩溃恢复成功率

### 运行时健康指标
- 熵增分数（系统状态"混乱程度"）
- 漂移分数（belief vs reality 的偏差量）
- 孤立 Worktree 数量
- 幻觉完成率（被验证推翻的 agent 自报完成）

### 人效指标
- 人工介入频率（越低越好）
- 人工审核单次耗时（越低越好）
- 每次人工介入后的循环收敛速度（越快越好）
- Sprint 吞吐量

## 12. 最终形态

Myrmidon 不是：**"AI 会写代码"**

Myrmidon 是：

> **一个长期自治运行、持续维护工程世界一致性的运行时系统**

其中：

- **Runtime** 维护世界状态，是唯一权威
- **Artifact** 保存现实，是唯一真相
- **Validator** 决定完成，包括自动化和人工
- **Reconciliation** 保持 belief 与 reality 一致
- **Worker** 是瞬时认知单元，生产产物，不拥有状态
- **Human** 是校准者，处理系统边界内无法自主决策的问题

最终形成：

```
Deterministic Runtime（确定性运行时）
    +
Probabilistic Intelligence（概率性智能）
    +
Continuous World Reconciliation（持续世界对账）
    +
Bounded Autonomy with Convergent Feedback（有界自治与收敛反馈）
```
